# Secure Radiosonde Uploads With WireGuard And FTP

This guide explains how to let a radiosonde receiver send files to this Skew-T server using legacy FTP without exposing FTP to the public internet.

The intended reader is comfortable on Linux and with editing configuration files, but does not need prior knowledge of VPNs, Docker networking, or FTP internals.

## What problem this solves

Many radiosonde receiver systems can only send data by FTP after a launch. Plain FTP is not secure enough to expose directly to the internet.

The approach documented here solves that by:

1. creating a private WireGuard VPN between the receiver computer and the server
2. running FTP only on the server's private VPN address
3. keeping the existing Skew-T application mostly unchanged

In practice, the receiver computer can be on a phone hotspot or any other internet connection, establish the VPN tunnel, and then send its FTP upload through that tunnel.

## High-level picture

There are three moving parts:

1. the Ubuntu host running this repository
2. the receiver-side computer connected to the radiosonde system
3. the Skew-T Docker stack

The data path is:

1. the receiver computer connects to the internet
2. it brings up a WireGuard tunnel to the Ubuntu host
3. it uploads a `*SHARPY.txt` file by FTP to the host's WireGuard IP
4. the FTP sidecar container writes that file into `data/work`
5. the existing processor in this repository detects the file and generates the image and web metadata

## Why this design is recommended

- FTP is never exposed on the public internet
- the application container does not need VPN privileges
- the host-side network boundary is clear and easier to debug
- the receiver can roam between networks, including a phone hotspot
- this matches the "legacy instrument + modern secure transport" problem cleanly

## What you need before you start

- an Ubuntu host that already runs this repository with Docker Compose
- a public DNS name or public IP for that host
- ability to open or forward one UDP port to that host, typically `51820/udp`
- ability to install a WireGuard client on the receiver-side computer
- a radiosonde receiver workflow that can do passive FTP uploads

Passive FTP is important. Active FTP is much more fragile when NAT, VPNs, and mobile hotspots are involved.

## A few terms

If VPN terminology is not familiar, these are the only ideas that matter here:

- `WireGuard`: a VPN that creates a private encrypted network link between two machines
- `peer`: one of the machines participating in the VPN
- `endpoint`: the public DNS name or IP and UDP port used to reach the VPN server
- `VPN IP`: the private IP address used inside the WireGuard tunnel
- `Docker Compose override`: an additional YAML file that adds or modifies services when you start the stack

You do not need deep knowledge of these technologies to use this workflow.

## Files generated by the helper script

The helper script in this repository is:

- `scripts/install_wireguard_host.sh`

On a real run, it does two kinds of work:

1. host-side WireGuard setup
2. repository-side FTP configuration generation

It writes these host files:

- `/etc/wireguard/wg0.conf`
- `/etc/wireguard/server_private.key`
- `/etc/wireguard/server_public.key`
- `/etc/wireguard/clients/<client-name>.conf`
- `/etc/wireguard/clients/<client-name>_private.key`
- `/etc/wireguard/clients/<client-name>_public.key`

It also writes these repository files:

- `config/wireguard-ftp.env`
- `docker-compose.ftp-wireguard.yml`

Checked-in examples of the generated repository files are available here:

- `config/wireguard-ftp.env.example`
- `docker-compose.ftp-wireguard.yml.example`

## Recommended default network values

Unless you already have a reason to choose something else, use these defaults:

- WireGuard interface: `wg0`
- WireGuard UDP port: `51820`
- host VPN IP: `10.44.0.1/24`
- receiver VPN IP: `10.44.0.2/32`
- FTP passive port range: `30000-30009`

These defaults are already built into the helper script.

## Step 1: Preview the configuration

From the repository root, run:

```bash
bash scripts/install_wireguard_host.sh \
  --endpoint your.host.example.org \
  --dry-run
```

What the dry run shows:

- packages that will be installed
- the host-side WireGuard files that will be written
- the repository-side FTP configuration files that will be written
- the rendered server WireGuard config
- the rendered client WireGuard config
- the rendered FTP env file
- the rendered Compose override file
- the exact real command to run afterwards

Important detail:

- the dry run works even if `wg` is not installed yet
- it will explicitly tell you that `wg` will be installed by the `wireguard` package

If you want non-default values, provide them now. Supported options are:

- `--endpoint HOSTNAME_OR_IP` required
- `--iface NAME` default `wg0`
- `--port PORT` default `51820`
- `--server-ip CIDR` default `10.44.0.1/24`
- `--client-ip CIDR` default `10.44.0.2/32`
- `--client-name NAME` default `receiver-laptop`
- `--ftp-user NAME` default `radiosonde`
- `--ftp-pass PASSWORD` default auto-generated on first real install
- `--ftp-pasv-min-port PORT` default `30000`
- `--ftp-pasv-max-port PORT` default `30009`

Example dry run with non-default values:

```bash
bash scripts/install_wireguard_host.sh \
  --endpoint your.host.example.org \
  --iface wg1 \
  --port 51821 \
  --server-ip 10.55.0.1/24 \
  --client-ip 10.55.0.2/32 \
  --client-name ground-station \
  --ftp-user radiosonde \
  --ftp-pasv-min-port 30100 \
  --ftp-pasv-max-port 30109 \
  --dry-run
```

## Step 2: Run the real install on the host

Once the dry run looks correct:

```bash
sudo bash scripts/install_wireguard_host.sh \
  --endpoint your.host.example.org
```

What this real run does:

- installs `wireguard` and `qrencode`
- creates the host WireGuard configuration
- generates server and client keys
- enables and starts the WireGuard interface
- writes the repository files needed by the FTP sidecar
- prints the generated client config location
- prints the FTP username and password

If you do not supply `--ftp-pass`, the script generates a password automatically the first time it runs.

If you run the script again later, it will reuse the previously generated FTP password from `config/wireguard-ftp.env` unless you explicitly pass a new one.

## Step 3: Make sure the network can reach WireGuard

The VPN itself uses one UDP port on the host, typically `51820/udp`.

You need to allow that traffic:

- in the host firewall
- in any cloud firewall or security group, if applicable
- in the home or office router, if the host sits behind NAT

For a typical Ubuntu host using `ufw`:

```bash
sudo ufw allow 51820/udp
```

If the host is behind a router, forward `51820/udp` from the router to the Ubuntu host.

For this use case, you do not need general IP forwarding or full site-to-site routing. The receiver only needs to reach services running on the host itself.

## Step 4: Configure the receiver-side computer

The helper script writes a client config on the host, typically:

- `/etc/wireguard/clients/receiver-laptop.conf`

That file contains everything the receiver-side computer needs for the VPN.

Install a WireGuard client on the receiver-side computer and load that client config there.

The client config will look like this:

```ini
[Interface]
Address = 10.44.0.2/32
PrivateKey = RECEIVER_PRIVATE_KEY
DNS = 1.1.1.1

[Peer]
PublicKey = SERVER_PUBLIC_KEY
Endpoint = your.host.example.org:51820
AllowedIPs = 10.44.0.1/32
PersistentKeepalive = 25
```

The most important line is:

- `AllowedIPs = 10.44.0.1/32`

That means only traffic to the server's VPN IP is sent through the tunnel. This keeps the VPN scope narrow and easy to reason about.

Bring up the tunnel on the receiver side and test it:

```bash
ping 10.44.0.1
```

On the host, you can check the peer status with:

```bash
sudo wg show
ip addr show wg0
```

## Step 5: Start the FTP sidecar

The helper script generates the files needed to make the FTP container match the host WireGuard settings:

- `config/wireguard-ftp.env`
- `docker-compose.ftp-wireguard.yml`

The env file contains:

- the host WireGuard IP
- the FTP bind IP
- the FTP passive address
- the FTP passive port range
- the FTP username
- the FTP password

The Compose override file adds an FTP container that:

- binds only to the host's WireGuard IP
- exposes FTP control plus passive FTP ports
- writes uploads into `./data/work`

Start the stack with the override:

```bash
docker compose -f docker-compose.yml -f docker-compose.ftp-wireguard.yml up -d
```

This is the key operational command after the host setup is complete.

## Step 6: Configure the radiosonde receiver FTP target

Use the values written into `config/wireguard-ftp.env`.

For the receiver's FTP settings:

- host: the `WIREGUARD_HOST_IP` value, typically `10.44.0.1`
- protocol: FTP
- username: the `FTP_USER` value
- password: the `FTP_PASS` value
- mode: passive
- remote directory: `/`

The FTP server should not be addressed by the host's public IP or public DNS name. It should be addressed by the private WireGuard IP.

## Step 7: Verify end-to-end

Once the VPN and FTP sidecar are up:

1. confirm the WireGuard tunnel is active
2. confirm the receiver-side computer can reach the FTP control port
3. upload a representative `*SHARPY.txt` file
4. confirm the file appears in `data/work`
5. confirm the existing processor handles it correctly

Useful commands:

On the host:

```bash
sudo wg show
docker compose ps
```

From the receiver-side computer:

```bash
nc -vz 10.44.0.1 21
```

To force immediate processing:

```bash
docker compose exec skewt /opt/venv/bin/python -m app.processor
```

To inspect the outputs:

- `data/output`
- `data/web/images`
- `data/web/sharpy`
- `data/web/manifest.json`

Then open the web UI and confirm the new sounding appears.

## Common operational questions

### Why not run WireGuard inside the application container?

Because that usually means extra privileges, more complicated network debugging, and tighter coupling between infrastructure and application logic.

The host is the correct place to define the network trust boundary.

### Why is FTP acceptable here at all?

Because FTP is not being exposed directly to the internet. It is carried inside the WireGuard tunnel, and the FTP service itself only listens on the WireGuard IP.

### Why use a separate FTP sidecar container?

It keeps the ingestion path isolated from the application container and makes the configuration easier to inspect and restart independently.

### Can I stage uploads instead of writing directly into `data/work`?

Yes. Edit the generated `docker-compose.ftp-wireguard.yml` and change the volume mount to something like:

```yaml
volumes:
  - ./data/work/incoming:/home/vsftpd/radiosonde
```

If you do that, make sure your operational workflow moves or copies files into the directory the processor actually scans.

## Troubleshooting

### The WireGuard tunnel does not come up

Check:

- the host's public DNS name or public IP is correct
- `51820/udp` is open on the host firewall
- `51820/udp` is forwarded by the router if the host is behind NAT
- the client config was copied correctly to the receiver-side computer
- `PersistentKeepalive = 25` is present in the client config

Useful commands:

```bash
sudo systemctl status wg-quick@wg0
sudo wg show
```

### The tunnel comes up, but FTP does not connect

Check:

- the FTP sidecar is running
- the FTP sidecar was started with the generated override file
- the receiver is configured for passive FTP
- the passive port range in `config/wireguard-ftp.env` matches the generated override
- the receiver is connecting to the WireGuard IP, not the public IP

Useful commands:

```bash
docker compose -f docker-compose.yml -f docker-compose.ftp-wireguard.yml ps
docker compose -f docker-compose.yml -f docker-compose.ftp-wireguard.yml logs ftp
```

### Files upload, but nothing appears on the website

Check:

- the uploaded filename ends with `SHARPY.txt` or `SHARPPY.txt`
- the filename contains `<SITECODE>_<YYYYMMDD>_<HH>Z_...`
- the file contains data between `%RAW%` and `%END%`
- the file is visible under `data/work`

Force a processing run:

```bash
docker compose exec skewt /opt/venv/bin/python -m app.processor
```

Then inspect:

```bash
docker compose logs skewt
```

## Suggested workflow for sharing with colleagues

If you want to share this method with other radiosonde station operators, the cleanest operational recipe is:

1. deploy this repository on an Ubuntu host with Docker
2. run the helper script in dry-run mode first
3. run the real install
4. copy the generated client WireGuard config to the receiver-side computer
5. start the Docker stack with the generated override
6. test one upload before relying on it operationally

That gives a repeatable pattern without asking colleagues to understand every detail of VPN or Docker internals.

## Quick reference checklist

Use this as the short version once you are familiar with the full guide.

### First-time setup checklist

1. Confirm the Ubuntu host has Docker Compose and a public DNS name or public IP.
2. Run:

```bash
bash scripts/install_wireguard_host.sh \
  --endpoint your.host.example.org \
  --dry-run
```

3. If the preview looks correct, run:

```bash
sudo bash scripts/install_wireguard_host.sh \
  --endpoint your.host.example.org
```

4. Open or forward `51820/udp` to the host.
5. Copy the generated client config from `/etc/wireguard/clients/` to the receiver-side computer.
6. Start the stack:

```bash
docker compose -f docker-compose.yml -f docker-compose.ftp-wireguard.yml up -d
```

7. Configure the receiver for passive FTP to the host WireGuard IP.
8. Upload one test file and verify it appears in the web UI.

### Per-launch operational checklist

1. Confirm the receiver-side WireGuard tunnel is connected.
2. Confirm the receiver FTP target is the host WireGuard IP, not the public IP.
3. Confirm the receiver is using passive FTP.
4. After launch, verify the uploaded file appears in `data/work`.
5. Verify the output appears in the UI.

### Quick troubleshooting checklist

1. No VPN:
   `sudo wg show`
2. VPN up, but no FTP:
   `docker compose -f docker-compose.yml -f docker-compose.ftp-wireguard.yml ps`
3. FTP upload succeeds, but no plot:
   `docker compose exec skewt /opt/venv/bin/python -m app.processor`
4. Still failing:
   `docker compose logs skewt`
   `docker compose -f docker-compose.yml -f docker-compose.ftp-wireguard.yml logs ftp`
